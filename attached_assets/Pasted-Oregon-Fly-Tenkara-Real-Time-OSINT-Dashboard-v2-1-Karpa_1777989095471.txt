1. Product Vision

The AI Fishing Buddy is not just a chat window. It is a living Oregon fishing companion.

* Live data = real-time nervous system
* Karpathy Wiki = long-term memory, fishing logs, preferences, river notes, spot intel, and learned patterns
* AI agent = brain that reasons over live data + personal knowledge
* Dashboard/map = simple interface for non-technical anglers

The user opens one link, checks the map, asks the Buddy where to fish, logs a trip, and the system remembers what worked.

AI

* OpenRouter unified API
* Model selector in sidebar:
    * Free/auto router
    * Fast model
    * Pro model
* OpenAI-compatible client
* App-side tool loop for function calling
* OpenRouter supports standardized tool calling across supported models, but the app must execute the tools and return results to the model.  ￼

Knowledge Base

Do not rely on local SQLite persistence in Streamlit Cloud production. Streamlit Community Cloud does not guarantee persistent local file storage, so local DB files can disappear after reboot/redeploy.  ￼

Use this split:

* SQLite = local dev, demo mode, export/import, offline backup
* Supabase Postgres or Turso/libSQL = production persistent memory
* FTS5 / keyword search = MVP retrieval
* Embeddings = optional semantic search after core app works

SQLite FTS5 is a built-in full-text search extension suitable for efficient keyword search over Wiki/log content.  ￼

5. Retrieval Logic

Every AI chat request should assemble context in this order:

1. User preferences
2. Recent fishing logs
3. Matching Wiki entries
4. Matching river/gage data
5. ODFW stocking context
6. Recent conversation history

Use hybrid retrieval:

* Exact river/spot/tag match
* Full-text search
* Optional embedding similarity
* Recency boost
* User preference boost
* Confidence/source boost

The AI should always explain recommendations in practical fishing language, not raw data language.

Example:

“The Crooked is in your tenkara comfort range, flows are stable, and your logs show you did well there last May on small dries. Fish soft seams and banks before noon.”

⸻

6. AI Tooling

Implement three app-side tools:

query_wiki

Searches preferences, logs, and Wiki entries.

update_wiki

Creates proposed structured updates.

Important: the model should not directly mutate trusted memory. The app validates the write first.

Auto-save allowed for:

* Fishing logs
* Chat summaries
* Low-risk notes

Confirmation required for:

* Preference changes
* Secret spot entries
* Deleting/editing existing knowledge
* Any safety-sensitive or legal guidance

get_live_data

Refreshes USGS/ODFW data on demand.

7. AI Behavior

System behavior:

You are a fun, witty, highly experienced Oregon fly/tenkara fishing buddy. You have access to live USGS/ODFW data and a private Karpathy Wiki containing user preferences, logs, and spot knowledge. Always use the Wiki and live data before answering. Give practical, safe, concise advice. Prefer actionable fishing guidance over raw data. Use light humor and fishing puns. If the user shares useful new fishing info, propose a structured Wiki update.

The Buddy should answer like:

* Where should I fish today?
* What flies should I bring?
* Is this river too high for tenkara?
* What did I do last time on the McKenzie?
* Log today’s trip
* Remember that this spot fishes best after rain
* Build me a Saturday plan within 90 minutes of Bend

⸻

8. Safety, Trust, and Provenance

Every recommendation should label its basis:

* Live: USGS/ODFW current data
* Logged: user’s own history
* Wiki: private saved knowledge
* Inferred: AI pattern recognition
* Unverified: user/forum/imported note

Add a “Why this recommendation?” expander that shows:

* Live data used
* Wiki entries used
* Log entries used
* Any uncertainty

11. Final Deployment Plan

1. Build locally with SQLite adapter.
2. Add production DB adapter for Supabase or Turso.
3. Store API keys in Streamlit secrets.
4. Deploy on Streamlit Cloud.
5. Use hosted DB for all persistent Wiki/log/preference data.
6. Keep SQLite export/import as backup and portability feature.

⸻

Final Dev Directive

Ship this as a living Oregon fishing memory system, not just a data dashboard.

The map shows what is happening now.
The live data explains current river conditions.
The Karpathy Wiki remembers the angler.
The AI Buddy combines all three into useful, personal, actionable fishing advice.

The key revision: SQLite is fine for local/offline/dev, but production memory must live in Supabase, Turso, or another persistent hosted database. Everything else in the original vision stands — just with safer memory, better retrieval, user separation, and auditable AI writes.
